## 1. Buggar & Kodkvalitetsproblem

### 1.1 Kritiska buggar

#### BUG: `sanitize_text()` prestanda-bugg (trust.rs:117-139)
- **Problem**: Loop som re-allokerar `.to_lowercase()` på *hela strängen* varje iteration, per pattern.
- **Worst case**: 41 patterns × N matchningar × full string lowercase = katastrofal prestanda på stora payloads (100KB+).
- **Fix**: Byt till single-pass med Aho-Corasick automaton, eller bygg manuell index-tracking.
- **Prioritet**: HÖG — detta är en hot path vid injection detection.

#### BUG: `.unwrap()` i produktionskod (server.rs:2433, 2475)
- **Problem**: Nästlad `.unwrap()` i `unwrap_or_else()` för HTTP header-parsning.
- **Fix**: Ersätt med `.expect("application/json is valid header")` eller säker fallback.
- **Prioritet**: LÅG — `"application/json"` är alltid giltig, men bryter mot kodstandarden.

#### BUG: `fetch.rs:29` — `RATE_LIMITERS` mutex `.unwrap_or_else(|e| e.into_inner())`
- **Problem**: Poison recovery på mutex. Korrekt hantering men bör loggas.
- **Prioritet**: LÅG — fungerar korrekt men döljer potentiella problem.

## 3. Latency-optimering — Topp 10

Sorterat efter estimerad påverkan (högst först):

### 3.1 Aho-Corasick för pattern matching (trust.rs)
- **Nuvarande**: 41 linjära `contains()`-sökningar per textnod = O(n×m)
- **Optimering**: Kompilera alla patterns till en Aho-Corasick automaton = O(n+m)
- **Estimerad vinst**: 5-10x snabbare injection scanning
- **Dependency**: `aho-corasick` crate (~50KB, ren Rust)

### 3.2 Återanvänd `reqwest::Client` (fetch.rs)
- **Nuvarande**: Ny `Client::builder().build()` per request (rad 63-75)
- **Optimering**: Global `OnceLock<Client>` eller acceptera Client som parameter
- **Estimerad vinst**: Eliminerar TLS-handshake + connection pool setup per request
- **Insats**: Liten

### 3.3 Parallellisera CSS-inlining (fetch.rs:254-261)
- **Nuvarande**: Sekventiell loop per CSS-fil
- **Optimering**: `futures::join_all()` / `tokio::join!()` för parallella downloads
- **Estimerad vinst**: 2-3x snabbare vid 3-4 CSS-filer
- **Insats**: Liten (tokio redan dependency)

### 3.4 Cacha lowercase goal i SemanticBuilder (semantic.rs)
- **Nuvarande**: `text_similarity()` anropas per nod, skapar nya Strings varje gång
- **Optimering**: Pre-compute `goal_lower` och `goal_words` en gång i `SemanticBuilder::new()`
- **Estimerad vinst**: ~300 allokeringar borta per typisk sida
- **Insats**: Minimal

### 3.5 HashSet istället för Vec::contains() (intent.rs, firewall.rs)
- **Nuvarande**: O(n) lookup per kontroll
- **Optimering**: `HashSet<&str>` / `HashSet<u32>` = O(1)
- **Estimerad vinst**: 2-5x snabbare i form-matching och firewall
- **Insats**: Minimal

### 3.6 Pre-allokera String-kapacitet (fetch.rs, parser.rs)
- **Nuvarande**: `String::new()` + upprepade `push_str()` = multipla reallokeringar
- **Optimering**: `String::with_capacity(estimated_size)`
- **Estimerad vinst**: ~15% färre allokeringar i parsing-pipeline
- **Insats**: Minimal

### 3.7 LazyLock för tracking domain HashSet (firewall.rs)
- **Nuvarande**: Linjär iteration genom 64+ statiska strängar + `format!()` per domän
- **Optimering**: `LazyLock<HashSet<&'static str>>` med prefix-matching
- **Estimerad vinst**: O(1) domän-lookup istället för O(64)
- **Insats**: Liten

### 3.8 Undvik `.to_lowercase()` i hot paths (63 anrop totalt)
- **Nuvarande**: 63 anrop till `.to_lowercase()` spritt över kodbasen
- **Optimering**: `to_ascii_lowercase()` för ASCII-jämförelser, cacha resultat
- **Estimerad vinst**: ~30% mindre allokering i parsing
- **Insats**: Medium (kräver audit av varje användning)

### 3.9 Streaming JSON-serialisering (lib.rs)
- **Nuvarande**: `serde_json::to_string()` allokerar hela JSON-strängen
- **Optimering**: `serde_json::to_writer()` med buffrad skrivning
- **Estimerad vinst**: ~10% snabbare för stora träd (300+ noder)
- **Insats**: Medium

### 3.10 Arena-allokering för SemanticTree
- **Nuvarande**: Varje SemanticNode allokeras separat på heapen
- **Optimering**: `bumpalo` arena — alla noder i ett kontiguöst minnesblock
- **Estimerad vinst**: 15-25% snabbare parse för stora sidor, bättre cache locality
- **Insats**: Stor (kräver livstids-annotationer)

## 4. Lightpanda-inspirerade förbättringar

### 4.1 `--obey_robots` som first-class flagga
- **Nuvarande**: `respect_robots_txt` finns i `FetchConfig` men är `false` som default och inte exponerad i MCP.
- **Åtgärd**: Lägg till `obey_robots: bool` i MCP-tools `fetch_parse`, `fetch_vision`, `fetch_click`, `fetch_extract`.
- **Insats**: ~2 timmar
- **Prioritet**: HÖG — trivial men viktigt för etisk positionering

### 4.2 `waitUntil: "networkidle0"` i CDP-backend
- **Nuvarande**: `vision_backend.rs` använder bara `tab.wait_until_navigated()` — väntar inte på XHR/fetch.
- **Problem**: Dynamiskt laddad CSS och data visas inte.
- **Åtgärd**: Implementera networkidle-logik via CDP Network events:
  1. Lyssna på `Network.requestWillBeSent` / `Network.loadingFinished`
  2. Vänta tills 0 aktiva requests i 500ms
  3. Fallback timeout 10s
- **Insats**: ~4 timmar (headless_chrome stödjer events)
- **Prioritet**: HÖG — enskilt största kvalitetsförbättring för screenshots

### 4.3 Per-steg timing i MCP-svar
- **Nuvarande**: Bara `inference_time_ms` och `preprocess_ms`. Fetch-tid finns i `FetchResult.fetch_time_ms` men skickas inte alltid tillbaka.
- **Åtgärd**: Lägg till i alla `fetch_*`-endpoints:
  ```json
  {
    "timing": {
      "fetch_ms": 150,
      "css_inline_ms": 45,
      "parse_ms": 2,
      "js_exec_ms": 0,
      "screenshot_ms": 35,
      "vision_ms": 40,
      "total_ms": 272
    }
  }
  ```
- **Insats**: ~3 timmar
- **Prioritet**: MEDIUM — kritiskt för profiling men inte funktionspåverkande

### 4.4 Inline XHR-logging i fetch_parse-svar
- **Nuvarande**: `detect_xhr_urls` kräver separat MCP-anrop.
- **Åtgärd**: Integrera `xhr_calls: [{url, method, status}]` direkt i `fetch_parse`-svaret när XHR-intercept är aktiverat.
- **Insats**: ~2 timmar
- **Prioritet**: MEDIUM

### 4.5 Verifiera VelloRenderer Send + Sync
- **Nuvarande**: Blitz-backend används i `vision_backend.rs` men parallellism ej verifierad.
- **Åtgärd**:
  1. Lägg till compile-time assert: `static_assertions::assert_impl_all!(VelloRenderer: Send, Sync);`
  2. Benchmarka 10 simultana Blitz-instanser
- **Insats**: ~1 timme
- **Prioritet**: HÖG — kritiskt för serverless scaling

### 4.6 Cookie-persistens
- **Nuvarande**: Cookies lagras bara i minnet. Går förlorade vid omstart.
- **Åtgärd**: Serialiserbar cookie-jar redan via `SessionManager` JSON. Exponera `session_export()` och `session_import()` för persistent storage.
- **Insats**: ~2 timmar
- **Prioritet**: LÅG — användaren äger state via JSON redan

## 5. Banbrytande idéer (utöver Lightpanda)

### 5.1 Speculative Pre-parsing (SPP)
- **Koncept**: När `compile_goal()` genererar en plan med flera steg, prefetcha *alla* sannolika next-page URLs parallellt.
- **Fördel**: Eliminerar sekventiell fetch-latency i multi-step flows.
- **Insats**: Medium — `orchestrator.rs` har redan `prefetch_suggestions`.

### 5.2 Inkrementell Parsing (Delta Parse)
- **Koncept**: Istället för att parsa om hela HTML:en, identifiera vilka DOM-noder som ändrats och uppdatera bara dem.
- **Fördel**: 10-100x snabbare för dynamiska sidor med små ändringar.
- **Insats**: Stor — kräver stabil nod-identitet mellan parseringar.

### 5.3 Compiled Relevance Index
- **Koncept**: Bygg en inverterad text-index (á la Tantivy) vid första parse. Efterföljande `text_similarity()`-anrop blir O(1) lookup.
- **Fördel**: Eliminerar all string-jämförelse i hot path.
- **Insats**: Medium

### 5.4 SIMD-accelererad Pattern Matching
- **Koncept**: Använd `memchr` crate (SIMD) för injection pattern scanning istället för `str::contains()`.
- **Fördel**: 4-8x snabbare substring-sökning via CPU SIMD-instruktioner.
- **Insats**: Liten — drop-in ersättning

### 5.5 Zero-Copy HTML Parsing
- **Koncept**: Undvik att kopiera HTML-strängen vid parsning. html5ever's Tendril är redan zero-copy internt, men vi kopierar vid `extract_text()`.
- **Fördel**: ~20% mindre allokering.
- **Insats**: Medium — kräver livstids-refactoring
