# RFC: stream_parse – Goal-Driven Adaptive DOM Streaming

**Branch:** `feature/stream-parse`  
**Typ:** New feature · Breaking API addition  
**Prioritet:** Hög – löser kontextbegränsning i fetch_parse  
**Status:** RFC – design klar, implementation pending  
**Datum:** 2026-03-19  
**Författare:** AetherAgent core team

---

## Sammanfattning

Idag levererar `fetch_parse` antingen hela DOM-trädet (1 286 noder på SVT.se – för mycket) eller `fetch_extract` med hårdkodade nycklar (15 tokens – kan missa relevant kontext). Det finns ingen mellannivå.

`stream_parse` löser detta med ett nytt MCP-verktyg som öppnar en SSE-stream mot en URL och låter LLM:en styra traversalen i realtid. Istället för att hämta allt och filtrera efteråt traverserar systemet DOM-trädet on-demand, emitterar de mest relevanta noderna först, och låter LLM:en bestämma när den har tillräckligt – eller vilken gren den vill expandera härnäst.

Resultatet är 13 noder istället för 1 286 för ett typiskt breaking-news-goal, med bibehållen förmåga att gå djupare på exakt de noder som är relevanta.

---

## Problembeskrivning

### Det nuvarande dilemmat

```
fetch_parse  → 1 286 noder → ~64 000 tokens → allt eller inget
fetch_extract → 15 tokens  → hårdkodade keys → kan missa kontext
```

Båda verktygen har fundamentala begränsningar:

`fetch_parse` traverserar hela DOM-trädet oavsett goal. En agent som vill ha breaking news på SVT exponeras för sportresultat, footer, cookie-inställningar och 1 270 andra noder den aldrig behöver. Dessutom finns ingen mekanism för att gå djupare på en specifik nod utan att köra om hela parse.

`fetch_extract` är det motsatta problemet. Den extraherar ett fast antal nycklar med confidence-scoring, men om en nod har låg confidence av fel anledning (t.ex. att "just nu" matchar en sidfot istället för en nyhetsbanner) finns ingen möjlighet att korrigera – resultatet är låst.

### Rotkorsaken

Båda verktygen är stateless request-response. LLM:en ber om något, får ett svar, begär nästa sak. Det finns ingen kanal för LLM:en att säga "stopp, det räcker" eller "expandera just den här grenen" medan parse pågår.

---

## Lösning

### Konceptet: Goal-Interrupt Streaming

Stream parsern vänder på kontrollen. Istället för att AetherAgent bestämmer vad som returneras bestämmer LLM:en det, i realtid, medan streamen pågår.

```
Utan stream_parse:
  URL → [hämta hela HTML] → [parse alla 1286 noder] → [filtrera] → leverera

Med stream_parse:
  URL → [parse chunk 1: top-10 relevanta noder] → emit → LLM evaluerar
       → directive: expand(nod 56)
       → [parse chunk 2: barn till nod 56, 3 noder] → emit → LLM evaluerar  
       → directive: stop
       → KLART – 13 noder totalt
```

Det unika är att LLM:en inte bara stänger streamen när den är nöjd – den kan styra **vilken gren** av DOM-trädet som expanderas härnäst, baserat på sin egen reasoning om goal.

---

## Arkitektur

Systemet är uppdelat i tre separata moduler med distinkta ansvarsområden. Separationen är avsiktlig – varje modul ska kunna bytas ut, testas och modifieras oberoende.

### Modul 1: State Manager

Ren sync-struct utan I/O-beroenden. Håller all mutable state för en pågående stream-session.

```rust
pub struct StreamState {
    pub sent_nodes:          HashSet<u32>,
    pub expanded_nodes:      HashSet<u32>,
    pub directive_queue:     VecDeque<Directive>,
    pub nodes_emitted:       usize,
    pub nodes_seen:          usize,
    pub relevance_threshold: f32,
    pub max_nodes:           usize,
}

impl StreamState {
    pub fn mark_sent(&mut self, id: u32)
    pub fn mark_expanded(&mut self, id: u32)
    pub fn push_directive(&mut self, d: Directive)
    pub fn next_directive(&mut self) -> Option<Directive>
    pub fn should_emit(&self, relevance: f32) -> bool
    pub fn checkpoint(&self) -> CheckpointPayload
    pub fn is_done(&self) -> bool
}
```

**Ansvar:** Aldrig emit samma nod två gånger. Hålla koll på vilka grenar som är traverserade. Serva nästa directive till Stream Engine. Beräkna `token_savings_ratio` live för checkpoint-events.

**Inte ansvar:** Inga futures, ingen async, ingen HTTP. Stream Engine och Decision Layer lånar `&mut StreamState`.

### Modul 2: Stream Engine

Hanterar all I/O – HTTP-fetch, chunked HTML-parsing, SSE-emission.

```rust
pub async fn run_stream(
    url:        String,
    state:      Arc<Mutex<StreamState>>,
    decision:   Arc<DecisionLayer>,
    sse_tx:     Sender<SseChunk>,
) -> Result<()>
```

**Ansvar:** Öppna HTTP/2-connection mot URL. Läsa HTML i chunks via `html5ever`. Leverera noder till Decision Layer för scoring. Skicka godkända noder som SSE-events. Reagera på directives från State Manager (expand, stop, next_branch). Hantera backpressure via TCP/tokio channel.

**Inte ansvar:** Vet ingenting om goal-scoring-logik. Vet ingenting om transport utanför sig själv.

### Modul 3: Decision Layer

Hanterar goal-scoring och routing av noder.

```rust
pub struct DecisionLayer {
    pub goal:    String,
    pub scorer:  RelevanceScorer,
}

impl DecisionLayer {
    pub fn score(&self, node: &SemanticNode) -> f32
    pub fn should_expand(&self, node: &SemanticNode, state: &StreamState) -> bool
    pub fn route(&self, node: SemanticNode, state: &mut StreamState) -> NodeDecision
}

pub enum NodeDecision {
    Emit,
    Prune,
    Queue { priority: f32 },
}
```

**Ansvar:** Beräkna relevance-score per nod baserat på goal. Besluta om en nod ska emittas, prunas, eller köas för eventuell expansion. Prioritera noder för emission baserat på score.

**Inte ansvar:** Vet ingenting om SSE. Vet ingenting om HTTP. Vet ingenting om State Manager direkt.

## MCP API

### Verktygsspecifikation

```
Namn:        stream_parse
Transport:   HTTP/2 + Server-Sent Events
Endpoint:    GET  /stream_parse   (SSE-stream nedåt)
             POST /directive      (LLM-directives uppåt)
```

### Request-parametrar

```json
{
  "url":               "https://www.svt.se",
  "goal":              "breaking news just nu",
  "top_n":             10,
  "min_relevance":     0.3,
  "max_nodes":         50,
  "vision":            false
}
```

| Parameter | Typ | Default | Beskrivning |
|-----------|-----|---------|-------------|
| `url` | string | – | URL att streama |
| `goal` | string | – | Agentens mål för goal-scoring |
| `top_n` | usize | 10 | Noder per chunk |
| `min_relevance` | f32 | 0.3 | Lägsta relevance för emission |
| `max_nodes` | usize | 50 | Hård gräns för totalt antal noder |
| `vision` | bool | false | Aktivera parallell vision-stream (fas 3) |

### SSE Event-typer (server → LLM)

**`nodes`** – emitterar en batch noder:
```
data: {
  "type":         "nodes",
  "chunk_id":     1,
  "nodes":        [...],
  "nodes_seen":   47,
  "nodes_total":  1286,
  "parse_ms":     1
}
```

**`checkpoint`** – progress-signal efter varje chunk:
```
data: {
  "type":                 "checkpoint",
  "chunk_id":             1,
  "nodes_emitted":        10,
  "nodes_seen":           47,
  "coverage_pct":         3.7,
  "token_savings_ratio":  0.78,
  "top_relevance":        0.95
}
```

**`done`** – streamen stängs normalt:
```
data: {
  "type":                 "done",
  "total_nodes_emitted":  13,
  "total_nodes_seen":     1286,
  "token_savings_ratio":  0.99,
  "parse_ms":             8
}
```

**`error`** – fel under streaming:
```
data: {
  "type":    "error",
  "code":    "fetch_failed",
  "message": "Connection refused"
}
```

### Directive-typer (LLM → server)

LLM skickar directives via `POST /directive` med `stream_id` från SSE-headern:

```json
{ "action": "expand",          "node_id": 56 }
{ "action": "stop" }
{ "action": "next_branch" }
{ "action": "lower_threshold", "value": 0.2 }
```

| Directive | Beskrivning |
|-----------|-------------|
| `expand(node_id)` | Expandera specifik nod – traversera och emittera dess barn |
| `stop` | Stäng streamen omedelbart |
| `next_branch` | Hoppa till nästa topprankade nod utan att traversera allt däremellan |
| `lower_threshold(value)` | Sänk `min_relevance` dynamiskt om för lite emitteras |

## Fullständigt flödesexempel

**Goal:** `"breaking news just nu"` på `https://www.svt.se`

```
Utan stream_parse: 1 286 noder → ~64 000 tokens
Med stream_parse:     13 noder →     52 tokens
```

**Steg 1 – Initial emission (Chunk 1):**
```
[Stream Engine hämtar HTML]
[Decision Layer scorar noder]
[State Manager: emit noder med relevance ≥ 0.3]

SSE emit →
  nod 56:  cta    · "Just nu: Vårdbiträde döms – misshandlade 96-åring"  · rel: 0.95
  nod 101: cta    · "Just nu: EU-ledarna övertygar ej Orbán om lånen"    · rel: 0.88
  nod 52:  main   · (container)                                           · rel: 0.10
  ...8 noder till...

Checkpoint: 10 noder emitterade / 47 sedda / coverage 3.7%
```

**Steg 2 – LLM-directive:**
```
LLM: "Jag vill ha mer om nod 56"

POST /directive → { "action": "expand", "node_id": 56 }

[State Manager: push expand(56) till directive_queue]
[Stream Engine: next iteration tar direktivet]
[Traverserar barn till nod 56]
```

**Steg 3 – Expansion (Chunk 2):**
```
SSE emit →
  nod 57: link · "/nyheter/lokalt/orebro/tva-till-sjukhus-..."  · rel: 0.80
  nod 58: text · "Misshandlade 96-åring på äldreboende"         · rel: 0.75
  nod 59: time · "2 minuter sedan"                              · rel: 0.60

Checkpoint: 13 noder totalt / 89 sedda / token_savings_ratio: 0.99
```

**Steg 4 – LLM stoppar:**
```
POST /directive → { "action": "stop" }

SSE emit →
  done: { total_nodes_emitted: 13, token_savings_ratio: 0.99, parse_ms: 8 }

[Stängd]
```

## Transport-strategi

### Fas 1 – SSE (nu)

SSE med separat POST-kanal för directives.

**Varför SSE räcker för fas 1:**

- HTTP/2 native – wreq-integrationen är redan på plats
- Backpressure gratis via TCP
- Debuggbar med `curl -N https://localhost/stream_parse?...`
- Passar "emit chunks"-mönstret perfekt
- `Last-Event-ID`-headern ger gratis reconnect med sessions-kontinuitet
- Finns i Axum/tower som inbyggt stöd

**Branch steering i realtid** är löst med SSE + POST. Round-trip är 1–5ms lokalt – detta är tillräckligt snabbt för DOM-traversal.

**Stateful sessions** är löst av SSE:s inbyggda `Last-Event-ID`. Om Cloudflare-tunneln tappar connection återupptas streamen från senaste checkpoint utan att börja om.

### Fas 3 – WebSocket (när vision + DOM kombineras)

WebSocket blir nödvändigt i exakt ett scenario: när `fetch_vision` och `stream_parse` kör parallellt mot samma sida och behöver dela samma connection.

SSE är text-only. PNG-frames från YOLO kan inte skickas i en SSE-stream utan base64-kodning, vilket ökar payload med ~33%. WebSocket hanterar binary frames nativt.

**Migreringsvägen:** Eftersom State Manager och Decision Layer är transport-agnostiska byts enbart Stream Engine-lagret ut. Inget annat i arkitekturen ändras.

WebSocket är relevant **när**:
- Aktiv branch steering baserat på live YOLO-detektioner krävs
- Flera simultana kontrollsignaler (DOM + vision) i samma session
- Directives kommer snabbare än ~1 per sekund

## State Manager – komplett datamodell

```rust
pub struct StreamState {
    // Vilka noder som redan skickats – O(1) lookup, aldrig dubbel-emit
    pub sent_nodes:          HashSet<u32>,

    // Vilka grenar som är fullt traverserade
    pub expanded_nodes:      HashSet<u32>,

    // Inkommande directives från LLM
    pub directive_queue:     VecDeque<Directive>,

    // Räknare för checkpoint och token_savings_ratio
    pub nodes_emitted:       usize,
    pub nodes_seen:          usize,

    // Justerbar runtime av LLM via lower_threshold-directive
    pub relevance_threshold: f32,

    // Hård gräns – stream stängs oavsett om LLM inte skickat stop
    pub max_nodes:           usize,
}
```

**Designprincip:** `StreamState` är ren sync. Inga `async fn`, inga futures, inga I/O-calls. Stream Engine håller en `Arc<Mutex<StreamState>>`. Decision Layer tar `&StreamState` (read) eller `&mut StreamState` (write) via explicit lån. Detta gör hela systemet trivialt att enhetstesta utan att starta en server.

**Estimat:** 5–7 dagar (beror på YOLO-träning)

## Öppna frågor för code review

**1. Directive-latens**
POST /directive är ett separat HTTP-anrop. I fas 1 med lokal server är round-trip 1–5ms. Är detta acceptabelt eller vill vi ha en snabbare kanalmekanism redan nu?

**2. max_nodes guard**
Förslaget har `max_nodes: 50` som hård gräns. Om LLM:en inte skickar `stop` och threshold är låg kan 50 noder nås snabbt. Ska vi istället ha en timer-baserad auto-stop?

**3. Concurrent sessions**
Ska en enskild server-instans kunna hantera parallella `stream_parse`-sessioner? Nuvarande design med `Arc<Mutex<StreamState>>` per session stödjer detta, men vi behöver en session-registry.

**4. Caching**
Om samma URL parsas med samma goal inom X sekunder, ska State Manager kunna returnera cached node-positions direkt? Relevant för multiagent-scenarion där flera agenter parsas samma sida.

**5. diff_trees-integration**
Stream parserns checkpoint-events innehåller `token_savings_ratio` och `nodes_emitted/seen`. Detta är kompatibelt med `diff_trees`-output-formatet (BUG-010). Om `diff_trees` exponeras via MCP kan stream_parse publicera checkpoints direkt till collab-store via `publish_collab_delta`.
